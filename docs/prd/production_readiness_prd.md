# PRD: Production Readiness Baseline

## Feature name

Production Readiness Baseline

## Status

Planned for first public beta and required before stable release.

## Summary

This PRD defines the baseline decisions and requirements Fastifly must satisfy before it can be considered production-ready for self-hosted users.

Fastifly will be an AGPLv3 self-hosted personal finance app with no telemetry by default, no email dependency, DB-backed jobs only, Docker Compose-first deployment, installable PWA support with limited offline command sync, manual production migrations, required backup support before beta, username/password plus passkey authentication, and a future Capacitor mobile path.

---

## Final baseline decisions

```text
License:              AGPLv3
Email:                no email support
Telemetry:            none by default
Queue:                DB-backed only
Offline v0.1:          installable PWA, limited safe command outbox
Backup:               required before beta
Migrations:            manual production migration command
Deployment:            Docker Compose first
Auth v0.1:             username/password + passkey
Mobile path:           Capacitor support, no React Native
```

---

## Goals

- Make Fastifly safe and practical for self-hosted production use.
- Keep deployment simple and low-infrastructure.
- Avoid mandatory third-party services.
- Protect the project with AGPLv3.
- Avoid telemetry by default.
- Support local authentication without email.
- Support passkeys from v0.1.
- Provide installable PWA support from early releases.
- Prepare for future Android/iOS apps using Capacitor.
- Require backup/restore support before public beta.
- Use manual production migrations to avoid unsafe automatic schema changes.

---

## Non-goals

- email verification

- SMTP configuration
- hosted SaaS account system
- mandatory telemetry
- Redis

- full offline-first/CRDT sync

- cloud backup provider integrations

## Email policy

### Decision

Fastifly will not require or support email in v0.1.

No SMTP.  
No email verification.  
No email invitations.  
No email password reset.

This keeps self-hosting simple and avoids external provider dependency.

```text

```

## Authentication

### Decision

v0.1 authentication will support:

```text
username/password
passkey
```

No email address is required for login.

### Username/password requirements

- Usernames must be unique.
- Usernames must be case-insensitive for lookup.
- Passwords must be hashed with Argon2id or another approved strong password hashing algorithm.
- Password policy should be reasonable, not hostile.
- Login attempts should be rate-limited.
- Sessions should use secure HttpOnly cookies.
- Production cookies must use `Secure=true`.
- Session expiration must be configurable.
- Logout must invalidate the active session.

### Passkey requirements

Passkeys should use WebAuthn.

Required capabilities:

- register passkey after account creation
- login with passkey
- list registered passkeys
- rename passkey
- remove passkey
- prevent removing the last usable auth method unless another auth method exists

### Recovery requirements without email

Because there is no email reset, Fastifly must support at least one safe recovery path.

Required for beta:

```text
admin CLI password reset
recovery codes
```

Recommended CLI examples:

```bash
fastifly user list
fastifly user reset-password <username>
fastifly user create-recovery-codes <username>
```

Recovery codes:

- generated by the app
- shown once
- stored hashed
- single-use
- revocable
- regenerating invalidates previous unused codes

### Acceptance criteria

- User can register with username/password.
- User can log in with username/password.
- User can register a passkey.
- User can log in with passkey.
- User can remove a passkey only if another login method remains.
- Admin can reset a password via CLI.
- Recovery codes work without email.
- Rate limiting protects login endpoints.
- Sessions use HttpOnly cookies.
- Logout invalidates session.

## Family/partner invitations without email

### Decision

Invitations must work without email delivery.

### Required flow

1. Owner/admin creates invite.
2. App generates a one-time invite link/token.
3. Owner/admin copies the invite link.
4. Owner/admin sends it manually through any channel.
5. Invitee opens link.
6. Invitee logs in or creates an account.
7. Invitee accepts invitation.
8. Membership is created.

### Invitation requirements

- Store token hash, not raw token.
- Invite tokens must expire.
- Invite tokens must be single-use.
- Owner/admin can revoke pending invite.
- Invite page must show workspace name and role.
- Invite acceptance must create audit event.
- App must not require an email address for invitation.

### Acceptance criteria

- Owner/admin can create copyable invite link.
- Invite can be accepted without email.
- Expired invite fails safely.
- Revoked invite fails safely.
- Accepted invite cannot be reused.
- Invite token is never stored raw.

```text
DB-backed jobs only
```

## PWA and offline behavior

### Decision

v0.1 will support:

```text
installable PWA
limited offline writes through device-scoped command outbox
```

### PWA requirements

- Web app manifest.
- App icons.
- Maskable icon.
- Service worker.
- Static asset precaching.
- Offline app shell.
- Offline status indicator.
- Update available prompt.
- Mobile responsive UI.
- Dark theme support.

### Offline v0.1 behavior

Allowed:

```text
app shell loads offline
offline message appears
possibly stale cached shell/assets
create simple expense/income/transfer transaction
create simple category
assign monthly category budget
queue approved commands in local outbox
sync approved commands when online
```

Not allowed offline:

```text
offline import commit
offline rule changes
offline recurring generation
offline reconciled-record updates
offline transaction delete/void
offline workspace/member/permission changes
offline exchange-rate changes
offline maintenance/correction commands
```

Every offline write must include:

```text
client-generated UUIDv7-compatible IDs
device_id
local_sequence
operation_id
operation_type
base_revision
idempotency_key
```

The server must replay offline operations through normal domain services, not raw table patches.

### Security requirements

- Do not blindly cache authenticated financial API responses.
- Do not cache session endpoints.
- Do not cache auth endpoints.
- Do not store secrets in localStorage.
- Logout should clear local app cache/state where appropriate.

### Acceptance criteria

- App is installable in supported browsers.
- App shell loads after first visit.
- App clearly says when offline.
- Approved offline writes are queued and visibly marked pending.
- Unsafe write actions are disabled or blocked when offline.
- Sync conflicts are explicit and reviewable.
- Sensitive API responses are not cached by the service worker by default.

## Backup and restore

### Decision

Backup support is required before public beta.

### Required backup scope

Must support:

```text
SQLite backup
SQLite restore
PostgreSQL backup documentation
PostgreSQL restore documentation
full JSON export
CSV export for core finance data
```

### SQLite backup requirements

- CLI backup command.
- Backup file includes app version and schema version metadata.
- Backup should be safe while app is running.
- Backup should warn if database is busy or locked.
- Backup should include clear restore instructions.

Recommended commands:

```bash
fastifly backup create
fastifly backup restore <file>
```

### PostgreSQL backup requirements

Use documented native PostgreSQL tooling.

Documentation must include:

```bash
pg_dump
pg_restore
```

or Docker Compose equivalents.

### Export requirements

Users must be able to export:

```text
accounts
transactions
categories
tags
budgets
currencies
exchange-rate snapshots
```

Formats:

```text
CSV for common tables
JSON for full structured export
```

### Restore requirements

- Restore must check app version compatibility.
- Restore must check schema version compatibility.
- Restore must warn before overwriting data.
- Restore must require explicit confirmation.
- Restore must be documented.

### Acceptance criteria

- SQLite user can create and restore backup.
- PostgreSQL user has working documented backup/restore steps.
- User can export transactions to CSV.
- User can create full JSON export.
- Backup is documented in README or docs/deployment.md.
- Migration docs recommend backup before production migration.

## Migrations

### Decision

Production migrations must be manual.

Development may use automatic migration if explicitly enabled.

### Requirements

- Separate SQLite and PostgreSQL migrations.
- Migration command must exist.
- Production startup must not silently modify schema.
- Production startup should fail with clear message if migrations are pending.
- Migration command should warn user to back up first.
- Migration status command should exist.

Recommended commands:

```bash
fastifly migrate status
fastifly migrate up
fastifly migrate down   # optional, only if safely supported
```

### Environment behavior

Development:

```text
AUTO_MIGRATE=true allowed
```

Production:

```text
AUTO_MIGRATE=false default
manual migration required
```

### Acceptance criteria

- Pending migrations are detected on startup.
- Production app does not auto-run migrations by default.
- Manual migration command works for SQLite.
- Manual migration command works for PostgreSQL.
- Migration docs include backup warning.
- CI tests migrations on both databases.

## Resolved baseline decisions

These decisions are fixed for v0.1/beta planning:

1. Usernames use slug-style login names; display names may contain spaces.
2. The first created user becomes instance admin.
3. Instance admin recovery powers are limited to auth/instance recovery, not silent cross-workspace data access.
4. Passkeys are optional but strongly encouraged during onboarding.
5. Generate 10 single-use recovery codes per user.
6. Invite links expire after 7 days by default.
7. App-generated backup encryption can wait; v0.1 documents external encryption and safe storage.
8. The Docker image includes the CLI.
9. Production startup fails clearly on pending migrations; `/ready` must stay unhealthy until migrations are handled.
10. Viewers cannot export by default. Export is an explicit permission/policy decision.
